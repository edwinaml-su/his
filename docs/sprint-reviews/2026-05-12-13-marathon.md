## 4. Incidentes y mitigaciones

### 4.1 Incidente Vercel persistente

**Síntoma**: deploys consecutivos fallaron por output directory (`apps/web/.next` vs `.next` relativo a rootDirectory) y workspace flag prisma generate.

**Root cause**: configuración de Vercel asume `outputDirectory` relativo al `rootDirectory` configurado en project settings, no al repo root. La documentación es ambigua.

**Mitigaciones aplicadas**:
- PR #5 (`ci(vercel)`): `--workspace` flag en prisma generate.
- PR #13 (`fix(vercel)`): outputDirectory = `.next`.
- Ajuste final en `vercel.json` ratificado por Edwin.

**Lección**: futuras integraciones Vercel deben verificar primero un deploy preview minimal antes de configurar build commands completos. Documentado en `docs/15_production_runbook.md` §3 (bootstrap).

### 4.2 Rotación triple de credenciales

**Síntoma**: 3 rotaciones de DATABASE_URL en una semana (rotación inicial + 2 emergencias).

**Root cause**: incidentes de exposición de password+PAT en sesiones anteriores (documentado en memoria sesión 1).

**Mitigaciones aplicadas**:
- `docs/15_production_runbook.md` §7 con matriz de rotación obligatoria.
- AUTH_SECRET, SUPABASE_SERVICE_ROLE_KEY, AUDIT_HASH_SECRET clasificados por cadencia mínima.
- AUDIT_HASH_SECRET marcado como inmutable (rotarlo rompe verificación cadena).

**Lección**: las 5 creds pendientes de rotación deben cerrarse antes del go-live (no bloqueante para esta sesión pero sí para G7).

### 4.3 Supabase MCP read-only

**Síntoma**: intentos de `apply_migration` fallaron por scope read-only del worktree de @Orq.

**Root cause**: por diseño — solo Edwin (humano) aplica migraciones a prod.

**Mitigaciones aplicadas**:
- @DBA entrega SQL listos para apply manual.
- `docs/21_db_operations.md` §5 documenta plan de apply paso a paso.
- @Orq no propone ni intenta apply automatizado.

**Lección**: separación de poderes correcta. Mantener.

## 5. Lo que mejorar (acción para próximo sprint)

1. **Smoke test pre-merge automático**. La inestabilidad Vercel se descubrió post-merge. Próximo: PR check obligatorio que despliega a preview + hits `/api/health` antes de permitir merge.
2. **ADRs co-creados con feature**. Hoy los ADRs llegaron en Fase 6; idealmente en Fase 4 junto con la implementación. Próximo sprint: PR template con sección "ADR justificación" obligatoria si introduce patrón nuevo.
3. **Advisor security en CI**. Hoy el chequeo es manual (MCP). Próximo: GitHub Action diario que ejecute `mcp__supabase__get_advisors` y abra issue automático si aparece CRITICAL.
4. **Cadencia de retro mini**. Esta sesión maratón cubrió 3 fases en 2 días sin retro intermedia. Próximo: retro corta de 30 min entre cada Wave/Stream para capturar lecciones en frío.
5. **Tag releases inmediatamente**. v0.1.0 fue tageada manualmente días después. Para v0.2.0 incluir el tag en el PR de release notes o hook post-merge.
